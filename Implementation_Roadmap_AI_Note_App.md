# 구현 로드맵 – AI Note App

## 1. 제품 방향

이 제품은 먼저 빠른 노트앱처럼 동작해야 하고, 그 다음에 AI 앱처럼 느껴져야 한다.

핵심 원칙:
- 첫 설치 용량은 가볍게 유지한다
- 저장은 AI 없이도 반드시 성공해야 한다
- AI는 로컬 저장 이후에만 실행한다
- 대형 온디바이스 모델은 첫 출시 필수 요소가 아니다

---

## 2. 앱 용량 전략

### 원칙
- 노트앱은 설치 시 무겁게 느껴지면 불리하다
- 사용자는 핵심 가치가 즉시 보일 때만 어느 정도 용량을 받아들인다
- 따라서 v1에서는 기본 앱에 대형 모델을 포함하지 않는 것이 좋다

### 권장 용량 기준
- v1 기본 앱은 가능한 한 작게 유지하고, 설치 체감 기준으로 100~150MB 이하를 목표로 한다
- 첫 스토어 빌드에 대형 모델은 넣지 않는다
- 먼저 fallback 로직으로 시작하고, 품질 향상 효과가 충분할 때만 소형 모델을 추가한다

### 결정
- v1: 대형 번들 모델 없음
- v1.5: 선택적 소형 온디바이스 모델
- v2: 지원 기기 대상 선택 다운로드형 AI pack

---

## 3. 출시 로드맵

### Phase 0: 기반 구축
목표:
- local-first 저장 구조를 먼저 검증한다

범위:
- Expo Prebuild + TypeScript + Zustand + SQLite
- Item, Folder, Sync Queue 스키마
- 로컬 트랜잭션 기반 저장 파이프라인
- 오프라인 우선 목록/상세 화면

성공 기준:
- 오프라인에서도 저장된다
- 저장 직후 항목이 즉시 보인다
- 저장 경로가 AI에 의존하지 않는다

### Phase 1: MVP 출시
목표:
- 결정적 규칙 기반 보강만으로도 충분히 쓸 만한 저장 앱을 출시한다

범위:
- URL/텍스트 입력
- Android Share Intent + iOS Share Extension 기본 구조
- 메타데이터 추출 Parse 레이어
- fallback 제목 생성
- fallback 3줄 요약
- fallback 썸네일 선택
- 폴더 지정
- 로컬 검색
- 백그라운드 sync queue

AI 전략:
- 대형 모델은 사용하지 않는다
- 규칙 기반 처리와 추출 로직만 사용한다
- `ai_status`는 `pending`, `completed`, `failed`를 지원한다

성공 기준:
- 입력부터 저장 완료까지 3초 이내
- AI 실패가 저장을 막지 않는다
- 오프라인 저장 후 나중에 동기화된다

### Phase 1.5: 경량 온디바이스 AI
목표:
- 앱이 무거워지지 않는 선에서 제목/요약 품질을 개선한다

범위:
- 교체 가능한 온디바이스 AI 인터페이스 추가
- 성능과 용량이 허용될 때만 소형 양자화 모델 1개 추가
- fallback 경로를 기본 안전장치로 유지
- 기기 성능 체크 추가

AI 전략:
- AI는 계속 비동기로만 동작한다
- 타임아웃은 3~5초를 유지한다
- 모델 로드에 실패하면 fallback 결과를 그대로 유지한다

성공 기준:
- fallback-only보다 제목/요약 품질이 개선된다
- 저장 속도는 느려지지 않는다
- 크래시나 메모리 압박이 허용 범위를 넘지 않는다

### Phase 2: 선택 다운로드형 AI Pack
목표:
- 모든 사용자에게 무거운 앱을 강제하지 않고 더 높은 품질의 AI를 제공한다

범위:
- 선택적 모델 다운로드 흐름
- 기기 성능/저장공간 적합성 체크
- 고급 온디바이스 AI 활성화 설정
- 모델 버전 관리

AI 전략:
- 기본 앱은 AI pack 없이도 충분히 사용 가능해야 한다
- 다운로드한 모델은 핵심 저장 흐름의 필수 요소가 아니라 데이터 자산으로 취급한다
- 스토어 심사용 리뷰 노트를 함께 준비한다

성공 기준:
- 기본 설치 용량은 계속 가볍다
- 고급 AI는 지원 기기에서만 동작한다
- 스토어 심사 리스크를 관리 가능한 수준으로 유지한다

### Phase 3: 공유 지식 제품 확장
목표:
- 개인 저장 앱에서 협업형 지식 저장 제품으로 확장한다

범위:
- 공유 폴더
- 멤버십 및 권한
- 향상된 충돌 처리
- Pro 구독

성공 기준:
- 공유 폴더 동기화가 안정적이다
- 로그인, 계정 삭제, 결제 정책까지 출시 준비가 완료된다

## 4. 단계별 기술 우선순위

### AI 전에 반드시 해결할 것
- 로컬 DB 스키마
- 저장 트랜잭션 안정성
- sync queue 구조
- Android/iOS 캡처 진입점

### 다운로드형 AI 전에 반드시 해결할 것
- 교체 가능한 AI 엔진 계약
- 백그라운드 job orchestration
- 기기 성능 체크
- 저장공간 관리와 정리

---

## 5. 아키텍처 결정

### 저장 경로
Capture -> Parse -> Local Save -> UI Update

### AI 경로
Saved Item -> AI Job Queue -> AI/Fallback -> Item Update

### 동기화 경로
Saved Item -> Sync Queue -> Server Sync

원칙:
- 저장 경로는 AI 경로를 절대 기다리면 안 된다

---

## 6. 권장 v1 포지셔닝

첫 출시는 이렇게 보여야 한다:
- 빠른 local-first 저장 앱
- 온디바이스 기반 스마트 보강 기능 포함
- 무거운 AI 어시스턴트 앱은 아님

이 방향이 중요한 이유:
- 사용자는 노트앱이 빠르게 설치되길 기대한다
- 사용자는 AI 품질이 조금 부족한 것보다 저장이 느린 것을 더 싫어한다
- 리텐션이 검증되기 전에는 대형 모델을 넣을 명분이 약하다

---

## 7. 실무 권장안

추천 순서:

1. 먼저 fallback-only MVP를 만든다
2. 저장 속도, 리텐션, 파싱 품질을 측정한다
3. 제목/요약 품질 개선 효과가 분명할 때만 소형 모델을 추가한다
4. 대형 모델 배포는 반복 사용성이 검증된 이후로 미룬다

현재 권장 결론:
- v1에 대형 모델은 넣지 않는다
- 앱 용량 자체를 제품 품질로 본다
- AI 품질 개선은 단계적으로 배포한다
